# VPS как мост и wdtt-server (концепция)

> **Статус:** проектирование, **не реализовано** в qWDTT/WDTT.  
> Описывает, как *должна* работать схема «один VPS = участник VK-звонка + VPN-сервер», если VK начнёт резать TURN на «левые» IP вне сессии звонка.

---

## Зачем это нужно

### Как сейчас (классический WDTT)

```
Телефон  ──►  TURN VK  ──►  peer = IP:порт VPS (DTLS)
                │
                └── для VK это «звонок, но трафик на чужой сервер в датацентре»
```

Оператор на телефоне видит **только VK** (белый список).  
VK на relay *может* ограничить peer: разрешать только **участников звонка** (ICE), а не произвольный VPS.

### Целевая схема (VPS = мост + сервер)

```
Телефон  ──►  TURN VK  ──►  peer = «участник звонка» на том же VPS
                                    │
                                    ▼  localhost / unix-socket
                              wdtt-server (DTLS, WireGuard)
```

Для VK трафик идёт **к собеседнику в том же join**, а не «куда-то в интернет».  
Физически это **одна машина**: звонковой leg и `wdtt-server` живут на VPS, между ними — локальная пересылка.

---

## Участники и роли

| Компонент | Где | Роль |
|-----------|-----|------|
| **qWDTT (телефон)** | Android, LTE с белым списком | VPN-клиент, TURN, WRAP/DTLS; peer **не** публичный IP VPS напрямую, а адрес **участника** из сессии |
| **wdtt-bridge** (новый) | VPS | Входит в VK-звонок (тот же hash), поднимает TURN/relay leg, принимает DTLS от телефона, **форвардит на localhost** |
| **wdtt-server** | VPS (тот же хост) | DTLS, WRAP, WireGuard, пароли — как сейчас |
| **VK TURN** | Инфраструктура VK | Relay только между «легитимными» peer в рамках звонка |

## Поток данных (пошагово)

### 1. Подготовка на VPS

1. Установлен `wdtt-server` (деплой из приложения или вручную).
2. Запущен **`wdtt-bridge`** (отдельный процесс или режим `wdtt-server`):
   - join в звонок по **тому же VK hash**, что и телефон;
   - получение TURN-кредов (anonymous API или сервисный аккаунт VK);
   - Allocate relay, публикация **ICE-кандидата** «участника B» в сессии (или использование relay-адреса как peer для A).

3. `wdtt-server` слушает, например:
   - `127.0.0.1:56000/udp` — DTLS (только локально);
   - или abstract socket / pipe между bridge и server.

### 2. Подключение телефона

1. Пользователь в qWDTT указывает профиль: hash, пароль, **режим «bridge»** (или auto-discovery peer от сервера).
2. Телефон получает TURN (свой hash / anonymous).
3. **Peer для TURN** — не `203.0.113.10:56000`, а адрес из **signaling/ICE** второго участника (relay VPS в звонке).
4. Пакеты: WRAP/RTP → TURN VK → **bridge на VPS** → unwrap → **DTLS на 127.0.0.1:56000** → `wdtt-server`.
5. WireGuard на телефоне поднимается как сейчас (GETCONF по DTLS).

### 3. Исход в интернет

Клиентский трафик после WG:

```
Телефон (WG) → … → wdtt-server → NAT → интернет
```

Опционально **WARP egress** ([deploy.sh](../app/src/main/assets/deploy.sh)) — только для выхода клиентов, не для leg VK.

## Схема на одном VPS

```
                    ┌─────────────────────────────────────────┐
  Телефон           │              VPS                         │
  (qWDTT)           │                                          │
      │             │  ┌──────────────┐    ┌───────────────┐  │
      │  TURN/WRAP  │  │ wdtt-bridge  │───►│ wdtt-server   │  │
      └─────────────┼─►│ (участник VK)│loop│ :56000 DTLS   │  │
         VK relay   │  │ TURN leg     │back│ :56001 WG int │  │
                    │  └──────────────┘    └───────┬───────┘  │
                    │                               │ NAT      │
                    └───────────────────────────────┼──────────┘
                                                    ▼
                                               интернет
```

**Важно:** снаружи VPS для VK — это **участник звонка**; порт DTLS наружу можно **не открывать** (только localhost), если весь транспорт идёт через TURN leg bridge.

## Сравнение с текущим режимом

| | Сейчас | VPS = мост + server |
|--|--------|---------------------|
| Peer TURN | Публичный IP VPS | Relay/ICE «участника» на VPS |
| DTLS снаружи | `0.0.0.0:56000` | Часто только `127.0.0.1` |
| Процессы на VPS | 1 (`wdtt-server`) | 2 (bridge + server) или 1 с двумя ролями |
| Вид для VK | Звонок → датацентр | Звонок → собеседник |
| В qWDTT | ✅ | ❌ не реализовано |

## Что нужно реализовать (чеклист)

### На VPS

- [ ] **`wdtt-bridge`** — Go-демон:
  - join по hash (переиспользовать `go_client/creds.go`, TURN из pion);
  - приём UDP от relay (от телефона), отправка на `127.0.0.1:56000`;
  - обратный путь: от `wdtt-server` → relay → телефон;
  - синхронизация hash/сессии с `wdtt-server` (config или bot API).
- [ ] **Сигналинг peer** — способ сообщить телефону «куда стучаться»:
  - вариант A: тот же `wdtt://`, но флаг `bridge=1` + bot отдаёт актуальный relay-адрес;
  - вариант B: телефон сам тянет ICE второго участника через расширенный API на VPS.
- [ ] **Systemd:** `wdtt-bridge.service` + `wdtt.service`, bridge стартует до server или вместе.
- [ ] **Один hash — два «участника»:** лимиты VK (квота relay, captcha) — мониторинг и fallback.

### В qWDTT (Android)

- [ ] Режим профиля **«VPS-bridge»** (peer dynamic / не IP VPS).
- [ ] Получение peer из bot/API или второго leg signaling.
- [ ] UI: пояснение, что на сервере должен крутиться bridge.
- [ ] Fallback на классический peer IP, если bridge недоступен.

### Операционка

- [ ] Логи: `[BRIDGE]` отдельно от `[СЕССИЯ]`.

## Ограничения и риски

1. **VK всё ещё видит аномалию** — долгий «звонок» без аудио/видео, большой объём UDP. Мост обходит *peer-whitelist*, не *behavior analysis*.
2. **Два leg на одном hash** — капча, «квота relay исчерпана», бан hash.
3. **Сложность** — два процесса, рассинхрон hash, обновление ICE при reconnect.
4. **Не заменяет белый список на телефоне** — телефон по-прежнему ходит только в VK TURN; выигрыш именно при **ограничении peer на relay**.
5. **Anonymous vs account** — на VPS bridge с anonymous API с того же IP datacenter может капчить чаще; аккаунт VK на bridge — отдельная политика.

## Когда имеет смысл

| Ситуация | Классический WDTT | VPS-bridge |
|----------|-------------------|------------|
| Сейчас всё подключается | ✅ достаточно | избыточно |
| WRAP timeout, TURN ok, peer режут | попробовать другой IP | ✅ имеет смысл |
| Только домашний интернет на VPS | residential IP проще | bridge не обязателен |
| Нужен минимум сущностей | 1 процесс | 2 процесса / 2 роли |
